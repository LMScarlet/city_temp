人类最高指令：
因为转写插件时你们AI经常误解TEST_MODE的用途，我明确说一下，它是“调试模式”而非“测试模式”更不是“测试缓存路径”，因为模型插件在实际业务中是要被外部C++服务平台调用并实例化的，而这会影响到os.getcwd()的结果，但是被外部调用的话我就没办法断点调试问题，所以才专门弄了这个TEST_BASE_DIR及开关来让调试模式下的base_dir指向外部C++服务平台路径，从而确保路径拼接结果一致来让程序正常运行（因为我不能让模型插件直接放在服务平台的路径里）。所以你们不要误解它的用途去瞎搞。

```python
from dmgis.common.dmsde import Rct

# 实例化 Rct() 时，底层会自动完成坐标极值的初始化（抹平了 C++ 中 _initrect 的需求）
rct = Rct() 

# 将空矩形框传入，底层会自动探测并回填实体（索引 0）的最大/最小经纬度
areawork.GetObjRect(0, rct)

```

* **排坑标记 (Marker)**:
在为 TIN 剖分（`calculate4pintzvalue`）约束边界，或为网格裁剪定位时，经常需要依赖目标区域的确切矩形。与提取多边形轮廓的 `GetObjPos` 原理一致，不要去拆解 `Se_Part` 手动计算最大最小值，直接调用 `GetObjRect(0, rct)` 让底层一次性吐出标准矩形即可，这在性能和精度上是唯一安全的做法。

### 【DMGIS 等值图模型 (Contour) Python 映射经验包】

**📌 [核心机制一：高级等值线/面生成管线]**

* **C++ 侧逻辑**:
利用 `grd_addcontour1` 逐个压入不同级别的阈值与属性，随后执行 `grd_pushcontour`，接着进行平滑 (`grd_setsleek`)、生成 (`grd_producecontourline`) 和闭合处理 (`grd_contourhandling`)。
* **Python 侧映射 (完全对应的原生 API)**:
```python
from dmgis.dtm.Grid import Grid

# grid 为 dmgis.dtm.Grid 实例
# 1. 压入等值线级别与对应的属性 (AInf/TInf)
grid.addcontour1(level.value, level.areaInf)
grid.addtextcontour(level.value, level.textInf)

# 2. 执行核心渲染管线 (严格保证顺序)
grid.pushcontour()                          # 压入准备
grid.setsleek(param.size)                   # 等值线平滑
grid.producecontourline(param.linetype, param.zls) # 生成等值线
grid.contourhandling()                      # 等值线闭合与后处理

```


* **排坑标记 (Marker)**:
调用上述管线前，`Grid` 实例必须已经通过 `grid.setgridwork(grid_work)` 挂载了物理网格数据。这套管线没有任何前缀，直接在 `Grid()` 实例上调用即可。

**📌 [核心机制二：将网格直接切片为矢量工作区 (SDA/SDL/SDT)]**

* **C++ 侧逻辑**:
`int ai = grd_getareawork(gridAnaly1, strSaveFileName, vecWorkShop);`
* **Python 侧映射**:
```python
# 必须传入目标保存路径和 VecWorkShop 工厂实例
ai_area = grid.getareawork(work_path + ".sda", vec_shop)
ai_line = grid.getlinework(work_path + ".sdl", vec_shop)
ai_text = grid.gettextwork(work_path + ".sdt", vec_shop)

if ai_area > -1:
    areawork = vec_shop.GetWorkByNo(ai_area)
    areawork.SaveWork(True) # 落盘保存

```


* **排坑标记 (Marker)**:
Python 端的 `getareawork` 等方法内部已经封装了 `SV_CreateAreaWork` 的逻辑。只需要把 `vec_shop` 作为第二个参数传进去，底层会自动在工厂中创建一个工作区并返回其 ID (`ai_area`)，极大简化了 C++ 中的繁琐调用。

**📌 [核心机制三：后处理绘图 (DrawByGraphics) 的 Pillow 降维重构标准]**

* **架构演进说明**:
针对在气象底图上叠加标题、站点文字、色块图例的需求，**绝对禁止**使用 `IS_Create()` 提取句柄并绑定 PyQt5 `QPainter` 的做法（极易引发内存段错误）。
* **Python 标准打法 (基于 PIL/Pillow)**:
参考 `dmgisplus_contour.py` 的实现方案，使用 `PIL.ImageDraw` 进行内存隔离的安全绘图：
```python
from PIL import Image, ImageDraw, ImageFont

# 1. dmgis 底层 Mapmsi 生成基础底图后 (例如 temp.png)
# 2. 使用纯 Python 的 Pillow 库重新加载图片
img = Image.open("temp.png")
draw = ImageDraw.Draw(img)
font_legend = ImageFont.truetype("msyh.ttc", 10) # 挂载字体

# 3. 使用标准 2D 绘图 API 绘制图例色块与文字 (安全且支持抗锯齿)
# 例：绘制矩形图例
draw.rectangle([x1, y1, x2, y2], fill=(r, g, b), outline=(0, 0, 0))
# 例：绘制极限值多边形图例
draw.polygon([(x1,y1), (x2,y2), (x3,y3)], fill=(r, g, b), outline=(0, 0, 0))
# 例：绘制文字
draw.text((x, y), "图例标题", fill=(0, 0, 0), font=font_legend)

# 4. 保存并覆盖原图
img.save("final_output.png", "PNG")
img.close()

```
建议内容：虽然使用 Pillow 降维重构的战略是对的，但如果在模型插件中需要 DMGIS 生成那张“初始地理底图”，切记在 mapDisp 绘图前，注入 FontSlib 等四个符号库并调用 InitData，否则底图也出不来。并且，用Pillow重构只允许用在非DMGIS类库接口的绘图逻辑上，比如纯用QT原生接口绘图，而如果用到了DMGIS专用类库接口绘图，就绝对禁止换用Pillow。

### 【DMGIS 雷电预报与点/面矢量处理 Python 映射补充经验包】

**📌 [补充机制一：矢量面工作区 (AreaWork) 的空间投影重构]**

* **C++ 侧签名**:
`WVA_PutMpara(areawork, &oldpa);`
`WVA_ProjTralateMap(areawork, &newpa);`
* **Python 侧映射**:
```python
# 必须的绝对引入路径
from dmgis.common.dmsde import Mpara

# areawork 为 VecWorkShop 获取到的面矢量工作区句柄
putmpa = areawork.PutMpara(old_para)
projmpa = areawork.ProjTralateMap(new_para)

```


* **排坑标记 (Marker)**:
之前经验包中提到的 `MapProj` 主要用于离散点坐标系的换算，而 `AreaWork` 对象自身已经封装了针对整个矢量图层的投影偏转管线。直接对工作区实例调用这两个方法即可完成整图扭曲，无需拆解多边形实体。

**📌 [补充机制二：点矢量工作区 (PntWork) 与点结构体 (PInf) 挂载流水线]**

* **C++ 侧签名**:
`_pinf inf; inf.Angle = 0; inf.col = 255; ...`
`_dt pt; pt.x = lon; pt.y = lat;`
`int index = WVP_AddObj(pntwork, &pt, &inf);`
* **Python 侧映射**:
```python
# 必须的绝对引入路径
from dmgis.common.dmsde import PInf, Dt

# 1. 创建空间点坐标
dt = Dt(lon, lat)

# 2. 组装点显示信息结构体 (注意 Python 侧颜色必须直接传十进制或 clr 对象)
pinf = PInf(
    subno = 57,
    Angle = 0.0,
    Height = 0.03,
    Wide = 0.03,
    col = clr_val, # 例如 255 或 16711680
    flag = 0
)

# 3. 压入点工作区
index = pnt_work.AddObj(dt, pinf)

```


* **排坑标记 (Marker)**:
这是构建 `.sdp`（点文件）的核心管线。参数传递时，`AddObj` 接收的必须是 `Dt` 对象和 `PInf` 结构体实例，这与面矢量工作区接收 `Se_Part` 和 `AInf` 的逻辑存在显著的类型隔离，严禁混用。

**📌 [补充机制三：矢量实体数量提取（逻辑数 vs 物理数辨析）]**

* **C++ 侧签名**:
`int num = WVP_GetObjNum(pntwork);`
`int numF = WVA_GetObjFNum(areawork);`
* **Python 侧映射**:
```python
num = pnt_work.GetObjNum()   # 获取面/点实体的逻辑数
num_f = pnt_work.GetObjFNum() # 获取面/点实体的物理数

```


* **排坑标记 (Marker)**:
**绝对死线！** 底层 DLL 对实体数量严格区分了“逻辑数”与“物理数”（通常与内部被标记为已删除但未物理抹除的内存有关）。在翻译 C++ 源码时，必须做到 **1:1 像素级复刻**：C++ 代码调用了 `GetObjNum`，Python 就必须写 `GetObjNum()`；C++ 调用了 `GetObjFNum`，Python 就写 `GetObjFNum()`。绝不可自行脑补替换，否则在遍历实体时会引发致命的越界段错误（Segment Fault）。

**📌 [补充机制四：矢量工作区落盘（覆盖保存与另存为的界限）]**

* **C++ 侧签名**:
`WVP_SaveWork(pntwork);` (保存当前修改)
`WVA_DownLoadToFile(areawork, areapath.toLocal8Bit().data());` (另存到新路径)
* **Python 侧映射**:
```python
# 场景 1：直接在已打开的源文件上覆盖保存修改
pnt_work.SaveWork() 

# 场景 2：将内存中的工作区切片另存为一个新的物理文件
areawork.DownloadToFile(areapath)

```


* **排坑标记 (Marker)**:
`PntWork` 与 `AreaWork` 的 `SaveWork()` 方法**不接收任何路径参数**。当需要覆盖原 `.sdp` 或 `.sda` 文件时（例如动态更新站点报警点位），直接调用无参的 `SaveWork()` 即可。只有当需要剥离出全新的外部文件时，才使用 `DownloadToFile(path)`。
