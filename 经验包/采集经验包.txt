* **排坑标记 (Marker)**:
1. **关于 `MapDisp` 和 `QPainterManager` 的避坑声明**：在常规的 DMGis 综合出图（即渲染整个工程）时，必须引入“三剑客”（Mapmsi, QPainterManager, MapDisp）并按序销毁。**但是！本雷达脚本的 C++ 源码走的是“单图层直出捷径”（调用了 `LYR_GetImage`）**，这直接跳过了画笔和渲染调度器。因此，在 Python 转写时，**绝对不需要**引入和实例化 `MapDisp` 和 `QPainterManager`，只需严格保证 `image.delete()` 的执行即可。
2. **内存黑洞预警**：`image.delete()` **必须放在 `finally` 块中确保执行**。如果 Python 因为中间的逻辑报错跳过了销毁，底层 DLL 内存将迅速溢出导致程序崩溃。

📌 [全局避坑底线：常驻服务 (Daemon) 下的 C++ 句柄显式释放法则]
现象误区：在编写单次运行的 Python 测试脚本（如 API 测试、画图 Demo）时，往往不需要手动释放 DmPrj 或其他底层句柄，因为 Python 进程结束后，操作系统会强制回收所有内存。这容易给开发者造成“不需要手动销毁”的假象。
致命隐患：本系统是一个 7x24 小时运行的后台常驻服务。在 withgis 架构中，由于采用了定时器+线程池（QTimer + QRunnable）的调度模式，每次定时任务触发时，都会重新 new 一个采集类实例（如 RadarCollector），并随之创建底层的 C++ 工程对象（如 DmPrj）。
底层真相：Python 的垃圾回收机制（GC）是基于引用计数的。对于 dmgis 这类经过封装的 C++ 库，Python 的 GC 只能看到外层一个极其轻量级的“壳对象”，它完全不知道底层 DLL 实际上在堆区开辟了多大的地图工程或位图内存。如果依赖 Python 的自动 GC 慢慢回收，C++ 底层的内存堆积速度将远超回收速度，最终引发服务器 OOM（内存溢出）崩溃。
铁律 (Marker)：凡是在定时循环或长生命周期内反复实例化的 DMGis 对象（特别是 DmPrj 工程句柄、Mapmsi 图像句柄、以及 Tin/Kig 等空间算法矩阵），在任务执行完毕后，必须无条件、显式地调用该对象的 delete() 或 Free() 方法！ 绝不能抱有“等 Python 自动回收”的侥幸心理。

```python
from dmgis.layer.DmPrj import DmPrj

prj = DmPrj()
```

### 📌 [API / 核心功能名称：AWX 云图文件挂载与边界探测]

* **C++ 侧签名**:

* **Python 侧映射**:

```python
# 【必须的绝对引入路径】
from dmgis.common.dmsde import Rct

# 1. 挂载本地 AWX 数据文件
cloud_layer.SetFlyFile(yt_path)

```
* **参数与类型交接**: 传入解压后或下载好的 `.AWX` 物理文件绝对路径。底层 `dmgis` 库会自动解析文件的空间属性。
* **排坑标记 (Marker)**: **层级调用变更预警！** 在 C++ 中，直接通过 `LY_GetRect(ytlayer, ...)` 就能拿到图层矩形范围；但在 Python 端的封装中（参考 `layer_test.py`），`FlyLayer` 自身并没有暴露出直接获取范围的方法。**转写员必须先调用 `GetWork()` 提取内部的工作区实例，再调用 `work.GetRect(rect)**` 才能成功回填空间边界数据。直接对 `cloud_layer` 调用获取矩形的方法会导致 `AttributeError` 崩溃。

### 📌 [API / 核心功能名称：投影参数组装 (兰伯特至经纬度)]

* **C++ 侧签名**:
`_mpara oldpara = _mpara(); oldpara.projType = PRJ_Lambert_Conformal_Conic; ...`
`_mpara newparajw = _mpara(); newparajw.projType = PRJ_LonLat; ...`
* **Python 侧映射**:
```python
# 【必须的绝对引入路径】
from dmgis.common.dmsde import Mpara
from dmgis.common.DmEnumType import ProjectionType, EarthParamType

# 构造云图原始投影 (兰伯特)
old_para = Mpara()
old_para.projType = ProjectionType.PRJ_Lambert_Conformal_Conic
old_para.earthParam = EarthParamType.EP_CGCS2000
old_para.lon = 110.0
old_para.lat = 30.0
old_para.lat1 = 30.0
old_para.lat2 = 60.0

# 构造输出投影 (经纬度)
new_para_jw = Mpara()
new_para_jw.projType = ProjectionType.PRJ_LonLat
new_para_jw.earthParam = EarthParamType.EP_CGCS2000
new_para_jw.dx = 0.0
new_para_jw.dy = 0.0

```
* **参数与类型交接**: 严格使用 `DmEnumType` 中的枚举值替代 C++ 的宏定义（如 `EP_CGCS2000`）。
* **排坑标记 (Marker)**: 虽然这里组装了投影参数，但**绝对不要**像雷达模块那样使用 `MapProj` 去做四个角点的极值换算 (`proj_to_lb.Convxy`)。云图的 C++ 源码中，直接采用了整图空间扭曲的策略。老老实实把这两个 `Mpara` 对象交给后面的 `Mapmsi` 去处理即可。

### 📌 [API / 核心功能名称：卫星影像渲染与空间扭曲纠偏 (单图层直出捷径)]

* **C++ 侧签名**:
`ISHAND image = LYY_GetImage(ytlayer, iwid, ihei, &rect, qRgb(0, 0, 0));`
`Is_PutMpara(image, &oldpara);`
`Is_ProjTralateMap(image, &newparajw);`
`Is_SavePng(image,savePath2.toLocal8Bit().data());`
`IS_Free(image);`
* **Python 侧映射**:
```python
# 【必须的绝对引入路径】
from dmgis.common.dmsde import clr

# 1. 按照比例计算出图高度 (必须防除零与空高度)
if rect.xmax == rect.xmin or rect.ymax == rect.ymin:
    return False
    
dscale = (rect.ymax - rect.ymin) / (rect.xmax - rect.xmin)
iwid = 1000
ihei = int(round(iwid * dscale))

# 2. 直接提取健全的 Mapmsi 图像实例 (切忌二次包装)
image = cloud_layer.GetImage(iwid, ihei, rect, clr(0x000000))

if image and hasattr(image, 'handle') and image.handle:
    try:
        image.PutMpara(old_para)
        image.ProjTralateMap(new_para_jw)
        image.SavePng(save_path2)
    finally:
        # 3. 强制释放底层内存
        image.delete()
```
参数与类型交接: 无论雷达层还是风云层，其 GetImage 方法返回的直接就是一个健全的 Mapmsi 对象实例。颜色参数必须使用 clr(0x000000) 表示纯黑透明底。
排坑标记 (Marker) 必须新增一条死线：严禁对图像句柄进行二次包装！ 绝对不能写出 Mapmsi(image.handle) 这种画蛇添足的代码。由于底层 C++ 胶水层已完成面向对象处理，强行二次包装不仅会导致运行时报错，还会剥夺底层的 _release 释放权。如果在 finally 块里调用 delete() 时底层认为自己没有释放权，将造成堆内存彻底泄漏，直接干穿服务器。

📌 **[API / 核心功能名称：网格独立调色板强制注入]**

* **C++ 侧签名**:
`double valset[7]{...};`
`_clr clrset[7]{...};`
`WDG_SetIntervalColorList(gridWork, 7, valset, clrset);`
* **Python 侧 / 第三方库映射**:
无需像 C++ 那样声明底层数组类型，Python 侧根据 `DtmGridWork` 的封装，直接传递标准 `list` 即可。
```python
# 【必须的绝对引入路径】
from dmgis.common.dmsde import clr

# 严格对齐 C++ 的硬编码数组范围
valset = [-2000.0, 0.0, 1000.0, 1500.0, 2000.0, 2500.0, 3000.0]
# C++ 的整数颜色必须用 clr() 包装
clrset = [clr(0), clr(16448250), clr(12110806), clr(8428986), clr(5013670), clr(7594692), clr(2529921)]

# 注入调色板
grid_work.SetIntervalColorList(7, valset, clrset)

```
* **参数与类型交接**: Python 端直接传原生 `list`，但列表内的元素必须严格是 `float` 和 `clr` 对象。底层会自动完成 `c_double` 等 ctypes 的转换。
* **排坑标记 (Marker)**: **底层数组越界红线**。传入的 `num` 参数（此处为 7）必须和 `valset` 与 `clrset` 列表的实际长度绝对相等！底层 DLL 函数缺乏高级语言的长度越界校验，一旦长度错位，会直接导致指针越界读取，引发 C++ 层面的 `Segment Fault` 进程秒崩。

📌 **[API / 核心功能名称：网格直出图形句柄与跨语言二次包装]**

* **C++ 侧签名**:
`ISHAND img = WDG_GetHImage(gridWork);`
`Is_SavePng(img, savePath2.toLocal8Bit().data());`
* **Python 侧 / 第三方库映射**:
`WDG_GetHImage` 返回的是 `c_void_p` 裸指针，Python 端必须引入 `Mapmsi` 进行手动包裹后，才能调用保存方法。
```python
# 【必须的绝对引入路径】
from dmgis.Mapmsi.Mapmsi import Mapmsi

# 1. 从网格中榨取底层图像指针 (返回 c_void_p)
img_handle = grid_work.GetHImage()

if img_handle:
    # 2. 核心操作：将裸指针传入 Mapmsi 构造函数进行二次包装
    # 注意：由于显式传入了 handle，Mapmsi 底层的 _release 会自动设为 False
    image = Mapmsi(handle=img_handle)

    # 3. 调用包装后的方法安全落盘
    image.SavePng(save_path2)

    # 【防雷死线】：绝对不要在这里调用 image.delete()！

```
* **参数与类型交接**: `grid_work.GetHImage()` 返回的裸指针直接作为关键字参数 `handle` 传递给 `Mapmsi` 构造函数。
* **排坑标记 (Marker)**: **内存所属权红线！** 在雷达和 FY2G 云图的业务中，我们使用的是 `cloud_layer.GetImage()`，该方法生成的图像内存是独立的，因此**必须**在 `finally` 中调用 `image.delete()` 释放；
但是！在 FY3D 业务中，图像是使用 `GetHImage()` 直接从 `grid_work` 内部提取出来的寄生对象。Python 端对其进行二次包装完成操作后，**绝对禁止**调用 `image.delete()`。该图像内存由 `grid_work` 托管，会随网格一起销毁。如果 Python 强行释放，将引发“Double Free（二次释放）”，导致网格引擎销毁时发生不可逆的内存崩溃。
